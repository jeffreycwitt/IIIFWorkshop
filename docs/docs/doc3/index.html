    Reflections on the Common Needs of an Image Viewer

    Exercise

    In this exercise, we want to build an organic bridge from the IIIF image API to the need for something like the IIIF image API.

    To get a feel for the need for this second API, I want us to think about ourselves as web application designers.

    If we wanted to build a photo gallery or image viewer for all the images we've now made available via the IIIF Image server.

    Since we don't have a lot of time to trouble shoot the creation of individual web pages.

    There are two ways we can do this, depending on your comfort level.

    1. Start making a simple html page(s) where you can insert images

    or

    1. Get out a piece of paper and pen and begin storyboarding your application.

    You could play around with your images like this: https://gist.github.com/jeffreycwitt/e95145c837488536d4802ee74b851353

    You can try this cropper tool to select regions of your images https://jbhoward-dublin.github.io/IIIF-imageManipulation/index.html?imageID=http://127.0.0.1:8182/iiif/2/demo.jpg

    Reflections

    As you proceed, I'd like you to consider the following questions:

    1. What classes of information (other than images) do you find yourself in need in order for you to get your application to work?

    2. What classes of relationships between classes of information does your application require?

    The IIIF Presentation API emerged out of a recognition that many image presentation applications need common classes of information and depend on common classes of information.

    But often this information is intertwined with the presentation logic of the application itself. Making it very difficult to re-use and application.

    But what if we could separate that data from the application.

    1. This would allow us to swap data in and out from the same application.
    2. This would allows us to swap viewers in and out for the same data.

    IIIF Presentation API Overview

    The basic object types of a IIIF 2.x manifest

    A common question: Why do we need a Canvas? Why isn't the image enough?

    We need the canvas abstraction, because there are many use cases where we might want to swap images in and out, while retaining the ability to associate annotations with the same coordinate space.

    See for example http://projectmirador.org/demo/advanced_features.html (check out the layers tab!)

    Creating a Manifest

    Method 1 (Bodleian manifest editor)

    https://iiif.bodleian.ox.ac.uk/manifest-editor/

    Create a new folder for all your manifests

    $ workshop $ mkdir manifests

    Export your manifest and save it in your "manifests"

    Method 2 (create a script)

    Once the basic anatomy of IIIF manifest is understood, most everyone will want to script the creation of their manifest. (No one wants to type hundreds of repetitive lines by hand.)

    This ruby script is an example of such automation. https://gist.github.com/jeffreycwitt/c3898946158ffdfff2d1c43f11666517

    To run, you need ruby installed and imagemagik (but this dependency can be circumvented by changing lines 53 and 57).

    It loops through a directory of images and will create a manifest for all the images in that directory.
